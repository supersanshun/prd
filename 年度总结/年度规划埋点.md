# 企业级年度个人统计埋点规划

## 1. 背景与目标

年度个人统计通常用于在年末或周期性节点，向员工、管理员或业务用户展示其在平台内的年度行为沉淀、效率成果、协作贡献与成长轨迹。该能力不仅服务于用户体验和品牌传播，也为运营、产品和管理层提供可复用的数据资产。

本规划目标：

1. 建立统一、可追溯、可扩展的“年度个人统计”埋点体系。
2. 支撑全员可见的年度报告入口、年度报告页、分享页、榜单/荣誉页、业务洞察后台等场景的数据消费。
3. 统一事件命名、字段口径、采集时机、质量校验与隐私治理要求。
4. 覆盖门户系统内所有核心模版的年度行为沉淀，确保报告具备完整业务代表性。
5. 为后续季度复盘、月度回顾、行业版报告等衍生能力预留扩展空间。

## 2. 适用范围

适用对象：

- 全体员工用户

入口策略：

- 系统内所有员工均可看到年度个人统计入口
- 不因角色、部门、岗位差异隐藏入口
- 若个别模版年度无数据，可展示“本年度暂未产生记录”，但不影响入口曝光和报告生成

覆盖端：

- Web 管理后台
- App 内嵌页面

覆盖阶段：

- 年度报告入口曝光
- 年度报告生成
- 年度报告浏览
- 卡片逐页浏览
- 分享、下载、导出
- 反馈、纠错、再生成

## 3. 业务问题拆解

年度个人统计主要回答以下问题：

1. 有多少用户看到了年度报告入口？
2. 有多少用户真正生成并查看了年度报告？
3. 门户系统内各业务模版在年度报告中的覆盖是否完整，是否存在模块缺失？
4. 报告内哪些模块最受关注，哪些页面流失最高？
5. 用户在员工考勤、文件通知、新闻公告、会议、站内信、OA流程、待办、消息、调查问卷、任务管理、日程等模版中的年度行为能否被准确还原？
6. 用户是否愿意分享，分享后带来多少二次访问？
7. 哪些核心指标最能体现个人价值与平台使用深度？
8. 数据生成链路是否稳定，是否存在缺数、错数、延迟问题？

## 4. 埋点设计原则

### 4.1 统一性

- 统一事件命名规范：`annual_personal_*`
- 统一用户标识、组织标识、页面标识、会话标识
- 统一时间口径与年度周期口径

### 4.2 分层设计

埋点分为四层：

1. 页面曝光层：页面、入口、弹窗、卡片曝光
2. 交互行为层：点击、滑动、切换、分享、下载
3. 业务结果层：报告生成成功、失败、重试、完成浏览
4. 数据质量层：数据加载成功率、接口耗时、字段缺失率

### 4.3 可复用性

- 年度埋点模型可兼容季度/月度总结
- 卡片页统一抽象为模块，避免每次单独重建
- 分享链路支持不同渠道扩展

### 4.4 可治理性

- 每个事件必须有明确触发时机、必填字段、校验规则
- 核心指标必须有唯一业务口径文档
- 上线前必须通过埋点联调和验收清单

## 5. 统计对象与核心指标

### 5.1 用户基础维度

- 用户 ID
- 用户类型
- 企业 ID
- 部门 ID
- 岗位/角色
- 入职时间
- 注册时间
- 活跃状态

### 5.2 门户模版覆盖清单

年度个人统计报告需覆盖门户系统内以下全部模版，并支持按模版输出独立卡片或聚合指标：

| 模版名称 | 建议模块编码 | 核心统计方向 |
| --- | --- | --- |
| 员工考勤 | `attendance` | 打卡天数、迟到早退、最早/最晚打卡 |
| 文件通知 | `file_notice` | 阅读次数、发布/接收次数、最近阅读时间 |
| 新闻公告 | `news_notice` | 浏览次数、阅读偏好、阅读高峰时间 |
| 会议 | `meeting` | 参会次数、发起次数、会议总时长 |
| 站内信 | `internal_mail` | 发送量、接收量、已读率、互动对象数 |
| OA流程 | `oa_process` | 发起流程数、审批数、通过率、平均处理时长 |
| 待办 | `todo` | 待办接收数、完成数、准时完成率 |
| 消息 | `message` | 消息接收量、点击量、消息响应时段 |
| 调查问卷 | `survey` | 参与次数、发起次数、完成率 |
| 任务管理 | `task` | 创建任务数、完成任务数、协作任务数 |
| 日程 | `schedule` | 创建日程数、参与日程数、最忙月份 |

埋点和数据产出要求：

- 以上模版全部纳入报告生成范围
- 若某个模版当年无数据，也要在生成链路中记录该模版已校验
- 报告生成成功事件需能说明模版覆盖数和缺失模版列表

### 5.3 年度核心指标建议

建议按五大类建设：

#### A. 活跃度指标

- 年度活跃天数
- 年度登录次数
- 月均活跃次数
- 首次访问时间
- 最后一次访问时间
- 最晚使用系统时间
- 最早使用系统时间
- 活跃高峰月份
- 活跃高峰时段

#### B. 效率产出指标

- 年度发起任务数
- 年度完成任务数
- 年度审批处理数
- 年度会议参与数
- 年度流程发起数
- 年度流程审批数
- 年度待办处理数
- 年度日程创建数
- 年度节省时长估算

#### C. 协作贡献指标

- 发送站内信数量
- 接收站内信数量
- 消息互动对象数

#### D. 模版行为指标

- 年度考勤打卡天数
- 年度文件通知阅读数
- 年度新闻公告阅读数
- 年度会议参与次数
- 年度消息接收量
- 年度问卷参与次数
- 年度任务完成数
- 年度日程参与次数

#### E. 成长荣誉指标

- 连续活跃最长天数
- 超越同部门用户百分位
- 年度高峰月份
- 个人关键词/标签
- 年度成就徽章数

### 5.4 报告展示指标示例

年度报告中的文案和卡片指标可参考以下表达方式：

- 今年你一共登录系统 `XX` 次，活跃了 `XX` 天
- 今年你最晚一次使用系统是在 `23:48`
- 今年你发送了 `XX` 封站内信，收到 `XX` 封站内信
- 今年你共参与 `XX` 次会议，累计会议时长 `XX` 小时
- 今年你处理了 `XX` 条待办，准时完成率 `XX%`
- 今年你发起了 `XX` 个 OA 流程，审批了 `XX` 个流程
- 今年你阅读了 `XX` 条新闻公告、`XX` 份文件通知
- 今年你完成了 `XX` 个任务，参与了 `XX` 次日程安排

以上示例要求对应到可落库、可校验、可复算的字段，不允许仅做前端静态展示。

## 9. 核心口径定义

为避免跨团队理解不一致，建议提前冻结以下口径：

1. 年度活跃天数
定义：用户在自然年内发生至少一次有效业务行为的去重天数。

2. 年度登录次数
定义：用户在自然年内成功登录门户系统的累计次数，同一次登录态续期不重复计数。

3. 最晚使用系统时间
定义：用户在自然年内最后一次有效业务行为发生的本地时间点，按用户所在时区展示。

4. 发送站内信数量
定义：用户在自然年内成功发送站内信的累计数量，不含草稿和发送失败记录。

5. 接收站内信数量
定义：用户在自然年内成功接收到的站内信累计数量，不含系统回收或撤回记录。

6. 完整阅读
定义：用户浏览到最后一页，且关键页停留达到设定阈值，例如总停留时长 >= 15 秒。

7. 分享成功
定义：收到客户端成功回执，或分享后落地页产生有效访问记录。

8. 生成成功
定义：报告核心模块均已完成数据装配，并返回可正常渲染的数据结构；门户 11 个模版均已完成数据校验并形成结果。

9. 有效曝光
定义：入口或页面进入视窗达到曝光阈值，例如停留 300ms 以上。

## 10. 数据漏斗设计

重点关注指标：

- 入口点击率 = 入口点击人数 / 入口曝光人数
- 生成成功率 = 生成成功人数 / 生成开始人数
- 打开率 = 报告曝光人数 / 入口点击人数
- 完读率 = 完整阅读人数 / 报告曝光人数
- 分享率 = 分享成功人数 / 报告曝光人数

## 11. 技术实现建议

### 11.1 采集方式

- 前端埋点：曝光、点击、页面切换、停留时长、分享操作
- 服务端埋点：报告生成、接口成功失败、数据装配耗时、模版覆盖校验结果
- 日志关联：通过 `request_id` / `report_id` 贯通前后端链路

## 13. 隐私与合规要求

- 不采集无必要的敏感个人信息
- 对外分享内容需经过脱敏和权限校验
- 用户排名、百分位等信息需确认是否允许展示
- 企业维度数据不可跨租户泄露
- 埋点日志需满足公司数据安全规范和存储周期要求

## 14. 角色分工建议

产品：

- 明确展示页面、统计指标、业务口径
- 输出埋点需求文档与验收标准

前端：

- 实现页面曝光、点击、切页、停留、分享埋点
- 保证弱网和页面切换下的上报稳定性

后端：

- 实现报告生成链路埋点
- 输出口径计算逻辑和接口字段说明

数据团队：

- 建立埋点表、宽表、主题看板
- 校验字段质量和业务一致性

测试：

- 执行埋点验收、异常链路测试、数据一致性测试

## 17. 推荐结论

对于企业级“年度个人统计”项目，建议优先围绕“入口触达 -> 报告生成 -> 浏览深度 -> 分享传播 -> 数据治理”五条主线建设埋点体系。先保证核心链路闭环，再逐步扩展荣誉标签、排行榜、个性化洞察等高级能力。这样既能满足业务展示需求，也能为后续年度运营与产品迭代沉淀长期可复用的数据基础设施。

本次方案需特别落实三项硬约束：一是系统所有员工都能看到年度入口；二是报告必须覆盖门户内员工考勤、文件通知、新闻公告、会议、站内信、OA流程、待办、消息、调查问卷、任务管理、日程共 11 个模版；三是报告指标必须尽量贴近用户可感知的年度行为，例如登录次数、发送/接收站内信数量、最晚使用系统时间等，避免只展示抽象聚合值。
