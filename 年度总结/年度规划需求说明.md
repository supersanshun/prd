# 年度规划需求说明

## 1. 文档说明

本文档用于说明“年度回顾”移动端原型的完整需求方案，面向产品、设计、前端、后端、测试与运营协作使用。

本文档覆盖内容：

- 功能说明
- 页面说明
- 展示字段说明
- 交互说明
- 文案逻辑说明
- 关键词生成说明
- 边界与验收建议

本文档不包含内容：

- 数据库表结构设计
- 物理表字段定义
- 接口协议细节
- 埋点表结构

## 2. 产品定位

### 2.1 产品目标

年度回顾不是单纯的数据罗列页面，而是一份面向员工个人的年度情感化回顾内容。核心目标是通过真实年度行为数据，向员工传递“被看见、被记录、被认可”的感受，同时兼顾企业关怀表达与分享传播价值。

### 2.2 核心价值

对员工：

- 看到自己一年的投入与成绩
- 获得企业表达的关怀感和认可感
- 形成可以分享和留存的年度纪念内容

对企业：

- 强化组织温度与员工体验
- 提升年度运营传播效果
- 激活员工分享意愿
- 沉淀可复用的年度回顾内容框架

### 2.3 产品风格要求

- 语气温暖，不做技术化表达
- 强调“陪伴、投入、坚持、被看见”
- 页面以叙事感为主，不以报表感为主
- 数据真实可信，但表达需要柔和

## 3. 适用对象与范围

### 3.1 适用对象

- 企业内全体员工

### 3.2 使用终端

- 移动端 H5
- App 内嵌页

### 3.3 使用场景

- 年末/年初个人年度回顾发放
- 企业文化传播
- 员工分享给同事或留存
- 年度活动入口承接页

## 4. 整体功能结构

当前原型由以下功能结构组成：

1. 入口页
2. 年度回顾正文页
3. 多页卡片式翻页浏览
4. 年度片段详情弹层
5. 分享海报弹层
6. 分享结果提示

页面结构采用“入口页 -> 多页叙事卡片 -> 弹层补充信息 -> 分享收尾”的路径。

## 5. 页面功能说明

## 5.1 入口页

### 功能目标

作为年度回顾的起始页，负责建立情绪氛围、展示员工基础信息、传达“这是一份专属年度回顾”的感受，并引导用户进入正文。

### 页面内容

1. 顶部主视觉
- 标题：`2025 年度回顾`
- 副文案：强调“记录认真投入的时刻”“整理成更有温度的回顾”

2. 氛围标签
- 示例：`送给每一位同事`
- 示例：`完整年度记录`
- 示例：`移动端回顾页`

3. 个人信息卡片
- 展示员工姓名
- 展示状态：`已准备好`
- 展示所属部门、岗位角色、回顾时间、生成时间

4. 入口摘要区
- 用一句概括性文案承接情绪
- 使用 3 个摘要数据强化年度感知

5. 主按钮
- 文案：`开启我的年度回顾`

### 页面价值

- 建立仪式感
- 明确“这是给我的”
- 在进入正文前先完成身份确认和情绪预热

## 5.2 年度回顾正文页

正文页采用分页叙事结构，当前原型共 8 页：

1. 封面页
2. 年度总览
3. 活跃轨迹
4. 协同沟通
5. 效率产出
6. 年度片段
7. 成长荣誉
8. 温暖收尾

每一页由以下内容中的一部分组成：

- 页标题
- 页副文案
- 重点数字
- 数据卡片
- 情绪化解释文案
- 时间节点说明
- 补充说明入口

## 5.3 年度片段详情弹层

### 功能目标

为员工提供更细的一层查看能力，但仍然保持轻量，不打断主叙事流程。

### 入口

- 点击底部按钮：`更多细节`
- 点击“年度片段”页中的任一片段卡片

### 弹层内容

分为两种模式：

1. 总览模式
- 展示“这一年的不同片段”
- 展示片段分类列表
- 展示回顾摘要

2. 单项模式
- 展示该片段的核心结论
- 展示年度指标
- 展示真实记录示例

### 页面价值

- 满足用户对细节的好奇
- 让页面内容更有真实感
- 强化“不是虚构文案，而是有依据的年度回看”

## 5.4 分享海报弹层

### 功能目标

承接用户的分享意愿，提供海报预览与多种后续动作。

### 展示内容

- 海报标题
- 员工姓名与部门
- 回顾时间
- 4 个核心摘要指标

### 当前原型支持动作

- 分享给同事
- 生成回顾长图
- 下载 PDF

### 页面价值

- 强化传播属性
- 支持员工留存自己的年度回顾
- 为后续分享能力接入提供展示框架

## 6. 各页功能详细说明

## 6.1 封面页

### 功能说明

封面页承担“致谢”和“定调”的作用，是正文第一页。通过一句感谢式标题和两段时间节点说明，传递企业对员工全年投入的认可。

### 主要内容

- 标题：`陈予安 的这一年，辛苦了`
  取数逻辑：取用户账户姓名，拼接固定文案“的这一年，辛苦了”。

- 副文案：说明回顾覆盖整个年度时间范围
  取数逻辑：取年度回顾起止日期，如 `2025.01.01 - 2025.12.31`，用于生成“这份回顾记录了你在全年里的每一次投入”类文案。

- 主数字：年度相伴次数
  取数逻辑：默认取用户一年内累计登录次数；若业务希望弱化频次感，也可切换为用户一年内累计使用系统天数，根据登录日志按天去重。

- 两个时间节点：
  - 这一年最早开始的一天
    取数逻辑：取用户一年内最早登录系统的时间。
  - 这一年最晚结束的一天
    取数逻辑：取用户一年内最后一次操作任一业务内容的时间，优先取有实际业务动作的时间，不仅取停留或打开时间。

### 呈现逻辑

- 标题不直接写“统计结果”
- 首先写“辛苦了”“谢谢你”
- 数字不直接作为结论，而作为“这一年的注脚”

## 6.2 年度总览页

### 功能说明

用于用最少的信息建立用户对全年节奏的整体认知，是全篇的总览页。

### 主要内容

- 相伴次数
  取数逻辑：取用户一年内成功登录次数累计值，基于登录成功日志统计。

- 投入天数
  取数逻辑：取用户一年内发生过有效行为的自然日天数，按天去重。有效行为包括登录、阅读、发送、审批、处理、创建、回复等。

- 最忙的时候
  取数逻辑：按月汇总用户年度行为总量，取综合活跃值最高的月份。综合活跃值可由登录、阅读、沟通、处理、创建等行为加权组成。

- 最专注的时段
  取数逻辑：按小时汇总用户全年行为发生次数，取行为总量最高的小时区间。

- 月度投入节奏柱状图
  取数逻辑：按月汇总用户每月的登录次数或综合行为次数，生成 12 个月趋势数组。

### 呈现逻辑

- 重点不是“你用了多少次”
- 重点是“你这一年的节奏稳定而踏实”
- 图表的存在用于增强真实感，不做复杂分析

## 6.3 活跃轨迹页

### 功能说明

展示员工一年的时间节奏特征，用“早开始、晚结束”的叙事体现投入强度。

### 主要内容

- 最晚仍在忙碌的时间
  取数逻辑：取用户一年内最后一次有效业务操作时间，优先取提交、审批、发送、完成、创建等有明确业务意义的动作时间。

- 最早开始的一次
  取数逻辑：取用户一年内最早一次登录或最早一次有效业务行为发生时间。

- 常投入时段热力概览
  取数逻辑：按小时段对用户全年行为进行聚合，输出多个高频时段及其强度值，用于生成热力块。

- 出勤相关摘要卡片
  取数逻辑：取员工考勤年度汇总数据，包括有效打卡天数、迟到次数、早退次数、最早打卡时间、最晚补签时间等。

### 呈现逻辑

- 用“时间”体现付出
- 用“清晨”和“深夜”体现画面感
- 用出勤摘要补充稳定性和持续性

## 6.4 协同沟通页

### 功能说明

展示员工一年的沟通、触达和连接能力，强调“回应”“协同”“被需要”。

### 主要内容

- 主动发出的沟通量
  取数逻辑：取用户一年内主动发送的站内信、通知、定向沟通消息等行为总量，按发送成功记录累计。

- 接收和响应的信息量
  取数逻辑：取用户一年内接收消息总量、点击查看总量、响应总量及响应时效数据。

- 文件阅读和公告关注情况
  取数逻辑：取用户一年内阅读文件通知、新闻公告的次数、阅读完成率、最近阅读时间及高峰月份。

- 问卷参与与发起情况
  取数逻辑：取用户一年内参与问卷次数、发起问卷次数、完成率、回收率等年度汇总值。

### 呈现逻辑

- 不强调技术动作
- 强调“连接了很多人，也回应了很多期待”
- 用主动发出和被看见构成双向关系

## 6.5 效率产出页

### 功能说明

展示员工在事项推进、协作、安排、会议、任务等方面的执行力。

### 主要内容

- 节省下来的时间
  取数逻辑：根据年度待处理闭环量、线上推进量、审批量、会议协同量等行为估算节省时长。该值为规则换算值，不是直接采集值。

- 事项推进情况
  取数逻辑：取用户一年内发起与处理的推进类事项数据，如 OA 流程发起数、审批数、通过率、平均处理时长。

- 待处理闭环情况
  取数逻辑：取用户一年内接收的待处理事项数、完成数、准时完成率、逾期数。

- 任务完成情况
  取数逻辑：取用户一年内创建任务数、完成任务数、协作任务数、准时完成率。

- 会议参与情况
  取数逻辑：取用户一年内参会次数、发起次数、累计会议时长、最长单次会议时长。

- 日程安排情况
  取数逻辑：取用户一年内创建日程数、参与日程数、最忙月份、最忙周安排数。

### 呈现逻辑

- 不强调“流程处理量”
- 强调“把复杂的事慢慢做成了”
- 节省时间不是系统效率宣传，而是对个人推动价值的侧写

## 6.6 年度片段页

### 功能说明

展示年度回顾的细分片段集合，让员工从不同类型的日常里看到自己的年度痕迹。

### 主要内容

- 片段总结 Banner
  取数逻辑：取年度片段配置数量和已成功汇总的片段数量，生成“这一年的点滴已整理完成”类总结文案。

- 11 个片段卡片
  取数逻辑：根据年度回顾配置的片段清单逐项生成，当前包含员工考勤、文件通知、新闻公告、会议、站内信、OA 流程、待办、消息、调查问卷、任务管理、日程。

- 每个卡片展示：
  - 片段名称
    取数逻辑：取该片段配置名称。
  - 片段主结论
    取数逻辑：取该片段最有代表性的一个年度聚合指标，拼接成一句主结论。例如考勤取有效打卡天数，会议取参会次数，站内信取发送与接收数量。

### 呈现逻辑

- 避免使用“模块”“模板”“数据分类”等技术词
- 统一用“片段”“记忆”“点滴”表达
- 引导用户点击查看更细内容

## 6.7 成长荣誉页

### 功能说明

用于总结员工这一年的个人特征，并通过关键词和状态描述形成“被理解”的感受。

### 主要内容

- 部门同频度
  取数逻辑：在同部门范围内，以年度活跃天数、沟通响应量、事项完成量、协同参与度等指标形成综合指数，再计算用户所处分位。

- 全员同频度
  取数逻辑：在全公司范围内，按与部门同频度相同的综合口径计算用户所处分位。

- 年度关键词
  取数逻辑：根据用户全年行为特征匹配关键词规则，如高频沟通、持续投入、夜间活跃、稳定推进等，输出 3 到 6 个关键词。

- 最长连续投入
  取数逻辑：按用户全年自然日行为记录，计算连续发生有效行为的最长连续天数。

- 最被需要的月份
  取数逻辑：按月汇总用户综合行为强度，取行为值最高的月份，结合沟通、处理、推进和协同数据生成。

### 呈现逻辑

- 不是评比排名页
- 不突出竞争氛围
- 更强调“你在团队中的存在感和稳定价值”

## 6.8 温暖收尾页

### 功能说明

作为结尾页，用感谢和总结收住全篇，并承接分享动作。

### 主要内容

- 一句总结性的感谢文案
  取数逻辑：基于用户年度最具代表性的 2 到 3 个核心指标自动生成总结句，通常取相伴次数、主动发出量、事项回应量等高代表性指标。

- 三个关键信息标签
  取数逻辑：从全年指标中选取最容易感知、最有记忆点的 3 个值，如最晚仍在忙碌时间、团队同频度、认真回应次数。

- 分享型 Banner
  取数逻辑：选取 1 条一句话总结文案和 1 组核心摘要指标，用于承接分享海报内容。

- 页面意义说明
  取数逻辑：该区域以固定文案为主，不依赖复杂取数，仅可根据企业是否开启“寄语”能力进行替换。

### 呈现逻辑

- 最后的感受应当是温柔、克制、真诚
- 不再增加复杂数据
- 把分享行为包装成留念和纪念，而不是传播任务

## 7. 展示字段说明

以下为页面显示层所需字段说明，不涉及数据库表字段设计。

## 7.1 基础信息字段

| 字段 | 含义 | 是否必需 | 说明 |
| --- | --- | --- | --- |
| `year` | 年度 | 是 | 回顾所属年份，如 2025 |
| `generatedAt` | 生成时间 | 是 | 回顾准备完成时间 |
| `company` | 企业名称 | 是 | 页面头部和海报使用 |
| `employee.name` | 员工姓名 | 是 | 个性化标题使用 |
| `employee.department` | 所属部门 | 是 | 入口页与海报展示 |
| `employee.role` | 岗位角色 | 是 | 入口页展示 |

## 7.2 年度摘要字段

| 字段 | 含义 | 是否必需 | 说明 |
| --- | --- | --- | --- |

| `summary.latestUse` | 最晚使用时间 | 是 | 用于“最晚仍在忙碌”表达 |
| `summary.earliestUse` | 最早使用时间 | 是 | 用于“最早开始的一次”表达 |
| `summary.peakMonth` | 高峰月份 | 是 | 用于“最忙的时候”文案 |
| `summary.peakHour` | 高峰时段 | 是 | 用于“最专注的时段”文案 |
| `summary.moduleCoverage` | 片段数量 | 是 | 当前用于展示已整理出的年度片段数量 |
| `summary.reportWindow` | 回顾时间范围 | 是 | 例如 `2025.01.01 - 2025.12.31` |
| `summary.savedHours` | 节省时间估算 | 否 | 用于效率页展示 |
| `summary.deptPercentile` | 部门同频度 | 否 | 用于成长荣誉页 |

| 字段 | 含义 | 是否必需 | 说明 |
| --- | --- | --- | --- |
| `label` | 时段标签 | 是 | 例如 `08 时` |

## 7.4 关键词字段

| 字段 | 含义 | 是否必需 | 说明 |
| --- | --- | --- | --- |
| `badges` | 年度关键词列表 | 是 | 展示为 3 到 6 个关键词标签 |

## 7.5 年度片段字段

页面中的“年度片段”本质上是不同业务场景的年度回顾项，但在用户侧不强调技术归类，只强调“不同维度的日常投入”。

### 片段对象字段

| 字段 | 含义 | 是否必需 | 说明 |
| --- | --- | --- | --- |
| `key` | 片段唯一标识 | 是 | 前端交互识别使用 |
| `token` | 片段简称 | 是 | 卡片首字标识 |
| `name` | 片段名称 | 是 | 例如员工考勤、会议等 |
| `hero` | 片段主结论 | 是 | 卡片最核心的一句话数据结论 |
| `summary` | 片段摘要 | 是 | 一句解释性描述 |
| `metrics` | 片段指标列表 | 是 | 弹层展示详细指标 |
| `records` | 真实记录示例 | 否 | 弹层展示时间型证据 |

### `metrics` 子项字段

| 字段 | 含义 | 是否必需 | 说明 |
| --- | --- | --- | --- |
| 指标名称 | 字段标签 | 是 | 如“有效打卡天数” |
| 指标值 | 字段值 | 是 | 如“231 天” |

## 10. 文案逻辑说明

## 10.1 文案设计原则

### 原则一：先情绪，后数据

先给员工一句能够建立情绪连接的话，再展示支撑这句话的数据。

错误方向：

- 你全年登录了 428 次

推荐方向：

- 这一年，你把许多普通的工作日，过成了值得被看见的一年
- 428 次相伴，246 天投入在线

### 原则二：少用技术词，多用感受词

避免使用：

- 模块
- 模板
- 覆盖率
- 处理量
- 活跃率
- 系统使用强度

优先使用：

- 片段
- 点滴
- 认真
- 投入
- 相伴
- 回应
- 被需要
- 被看见

### 原则三：数字要真实，但结论要温柔

数据必须有依据，文案不能夸张，但表达可以更有人情味。

例如：

- `最晚使用时间 23:48`

推荐写法：

- `23:48，你仍在把手头的事认真收好`

### 原则四：不做“绩效评语”，做“年度回望”

页面不是绩效考核结论，也不是员工评价结论，不应出现过强的管理色彩。

避免：

- 你的执行力很强
- 你的表现超过大多数同事

推荐：

- 你在团队中的持续投入和稳定回应，让很多事情更顺畅地发生

## 10.2 文案结构建议

每页文案建议遵循以下结构：

1. 情绪性标题
2. 一句解释性副文案
3. 核心数字
4. 一句“数字背后的意义”

## 10.3 数据字段到文案字段的转换逻辑

| 原始展示含义 | 用户侧表达建议 |
| --- | --- |
| 登录次数 | 相伴次数 |
| 活跃天数 | 投入天数 |
| 最晚使用时间 | 最晚仍在忙碌 |
| 最早使用时间 | 最早开始的一次 |
| 峰值月份 | 最忙的时候 / 最被需要的月份 |
| 峰值时段 | 最专注的时段 |
| 流程审批量 | 持续推进 / 认真回应 |
| 待办处理量 | 认真回应 / 事项完成 |
| 阅读数量 | 关注与了解 |
| 消息响应 | 及时回应 |

## 11. 关键词说明

## 11.1 关键词的目标

关键词用于概括员工这一年的气质和工作状态，不是标签化评价，而是让员工在最后一页感受到“企业看到了我这一年的样子”。

## 11.2 关键词的输出要求

- 每人输出 3 到 6 个关键词
- 关键词长度建议 4 到 7 个字
- 关键词应当积极、克制、可被接受
- 不使用负向或带考核意味的词

## 11.3 关键词风格要求

推荐风格：

- 高频协同者
- 稳步推进者
- 夜间守护者
- 始终在线的人
- 温和回应者
- 默默可靠的人
- 长线投入者
- 节奏稳定者

不推荐风格：

- 加班狂人
- 卷王
- 效率机器
- 审批达人
- 任务狂魔

## 11.4 关键词生成逻辑建议

关键词应根据年度真实行为组合生成，不能随机输出。

建议逻辑：

1. 活跃节奏类
- 活跃天数高
- 跨月持续稳定
- 最早/最晚时间跨度大

可映射关键词：

- 始终在线的人
- 长线投入者
- 节奏稳定者

2. 协同沟通类
- 主动发出量高
- 接收与响应量高
- 涉及对象数多

可映射关键词：

- 高频协同者
- 温和回应者
- 被需要的人

3. 推进执行类
- 待处理完成度高
- 推进类数据高
- 持续闭环稳定

可映射关键词：

- 稳步推进者
- 可靠执行者
- 把事做成的人

4. 特殊节奏类
- 夜间活跃明显
- 高峰期承担较多

可映射关键词：

- 夜间守护者
- 高峰接住的人

## 11.5 关键词使用规则

- 同一员工关键词不应自相矛盾
- 页面最终展示的关键词不宜过多
- 关键词需与页面前文数据一致
- 关键词应优先服务情绪收束，而非解释全部数据

## 12. 视觉与体验要求

### 12.1 整体体验

- 视觉风格温暖、轻盈、有节奏
- 页面应有仪式感和纪念感
- 不应出现后台报表风格

### 12.2 数据呈现要求

- 每页保留 1 到 3 个最重要数字
- 数字周围必须配有解释性文案
- 不堆砌过多指标

### 12.3 交互体验要求

- 翻页顺畅
- 触控区域清晰
- 弹层关闭方式明确
- 分享动作反馈明确

## 13. 异常与边界情况说明

## 13.1 数据不足

如果员工某些年度片段数据较少，应遵循以下原则：

- 页面仍可生成
- 文案改为轻量表达
- 避免出现“无数据”造成挫败感

推荐表达：

- 这一部分的记录不多，但也构成了你这一年的一小段注脚

不推荐表达：

- 该模块无数据

## 13.2 关键字段缺失

如姓名、部门、年度范围、核心数字缺失：

- 不允许直接上线展示
- 必须提供兜底文案或隐藏该区域

## 13.3 分享动作失败

- 需给予明确提示
- 不可无反馈

推荐提示：

- 当前暂未完成分享，请稍后再试

## 13.4 关键词无法生成

- 应提供兜底关键词
- 兜底关键词仍需保持温和

推荐兜底：

- 认真同行的人
- 这一年值得被记住的人

## 14. 验收建议

## 14.1 功能验收

- 入口页可正常进入正文
- 正文页可正常翻页
- 进度点可跳转
- 滑动翻页生效
- 年度细节弹层可打开关闭
- 分享弹层可打开关闭
- 分享动作有提示反馈

## 14.2 内容验收

- 所有核心数字均有对应解释文案
- 页面无“模板、模块、覆盖率”等技术化词汇暴露
- 全文语气统一温和
- 关键词与实际数据表现一致

## 14.3 体验验收

- 移动端阅读无压迫感
- 文字长度适中
- 重点数字层级清晰
- 整体读完后有“被看见”的感受

## 15. 后续扩展建议

后续可在当前原型基础上继续扩展：

1. 接入真实接口，替换静态示例数据
2. 增加更多个性化文案分支
3. 增加不同员工类型的回顾模板
4. 增加年度长图分享成品页
5. 增加企业寄语或直属领导寄语
6. 增加音乐、动效和翻页过渡效果

## 16. 总结

本需求的核心不是做一个“年度数据统计页”，而是做一份“员工年度回望内容”。页面中的数字只是依据，真正需要传递的是员工一整年的投入感、价值感和被看见的感受。

因此，整个产品设计必须坚持以下原则：

- 数据真实
- 文案温暖
- 交互轻量
- 情绪完整
- 分享自然

最终目标是让员工在看完整份回顾后，感受到一句明确但不过度渲染的话：

`谢谢你，这一年辛苦了。`
